Class and ID: Best Practices Towards Writing Your Best Code

09 Sept 2021 | 5 mins read | Tech

Two of the most widely used selectors for styling targeted HTML elements are classes and IDs. However, knowing when and how to use them can be quite challenging sometimes. While it is established that information which can be used multiple times should be kept in a class, data that is unique should be kept in an ID.

Notice, though, their fundamental distinction:

Class Attribute ID Attribute
used to specify a class for an HTML element used to specify a unique ID for an HTML element
multiple HTML elements can share the same class you cannot have more than one element with the same ID in an HTML document

Some would argue that IDs and classes can be used interchangeably if this means doing so will produce the result you want. To a certain extent, this is true. There is nothing you can do with an ID that you cannot do with class, and vice-versa. In fact, CSS does not really care which one you use.

But the principles laid out above are no arbitrary rules randomly formulated to make life difficult. Other programming languages do care, particularly JavaScript, which greatly depends on the ID selector to be able to execute certain functions. Also, these rules form part of the generally accepted protocols and guidelines that must be observed to ensure code validation and, overall, the continuous long-term growth of the web.

Here, I have come up with some of the best practices when using the CSS selectors class and ID so you can produce better outcomes when writing your best code yet.

Minimalism. The use of these 2 selectors should be kept to a minimum and should mainly be to serve styling or visual, not structural, purposes. Plainly, if you do not need them, then do not use them. Using these attributes judiciously and semantically means avoiding this:

Brevity. While there is no maximum limit for characters in CSS names, there is, however, a practical limit, especially if you want your code to render properly in different browsers. Ideally, though, names should be as short as possible, but they can be as long as necessary. For instance, a CSS file with 30 classes of long names can add up and result in a huge performance issue on a large site. Also, consider how important bandwidth is when using the web on a mobile device. Heaps of long CSS names can slow down download of content, and that is inconsistent with good user experience.

At the end of the day, HTML will execute and render your code no matter how bad it is. The practices above are but standard ways of doing things that are generally observed because they are efficient, produce desirable results, and minimise the unpredictability of non-compliant codes.
